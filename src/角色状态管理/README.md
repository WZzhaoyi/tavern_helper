# 角色状态管理脚本

一个简洁的角色状态管理框架，每个角色可以有一个或多个状态变量，每个状态在不同数值区间有不同content。

## 功能特性

### 1. 自动解析状态定义
- **从 prompt 中提取**：使用 `<character_states>` 标签在 prompt 中定义状态
- **支持多个状态**：每个角色可以有一个或多个状态变量
- **数值区间content**：每个状态在不同数值区间可以有不同的content
- **自动初始化**：新角色初始化为0，已有角色跳过初始化

### 2. 状态更新
- **LLM API 格式触发**：状态更新由 LLM API 输出中的 `_.set("角色名.状态名",oldvalue,newvalue)` 格式触发
- **支持多个更新**：一次可以更新多个状态
- **自动应用**：脚本自动解析并应用状态更新

### 3. Prompt 变量替换
- **状态注入**：在 prompt 发送前自动注入当前状态信息
- **区间匹配**：根据状态值自动匹配对应数值区间的content

## 使用方法

### 1. 在 Prompt 中定义状态

在角色描述、世界书或其他 prompt 位置使用 `<character_states>` 标签定义状态：

```json
<character_states>
{
  "name": "character_states",
  "characters": [
    {
      "name": "角色名",
      "states": [
        {
          "name": "好感度",
          "ranges": [
            {"min": 0, "max": 30, "content": "角色对用户态度冷淡，说话语气生硬"},
            {"min": 31, "max": 70, "content": "角色对用户态度正常，保持礼貌"},
            {"min": 71, "max": 100, "content": "角色对用户态度友好，语气亲切"}
          ]
        },
        {
          "name": "信任度",
          "ranges": [
            {"min": 0, "max": 50, "content": "角色对用户缺乏信任，不愿分享秘密"},
            {"min": 51, "max": 100, "content": "角色对用户非常信任，愿意分享秘密"}
          ]
        }
      ]
    }
  ]
}
</character_states>
```

**注意**：
- 每个角色可以定义多个状态
- 每个状态可以有多个数值区间
- 每个区间需要定义 min、max 和 content
- 状态值默认为数值类型，初始值为0

### 2. 配置脚本设置

在脚本设置界面中：
1. 启用"角色状态管理"
2. 启用"在聊天末尾注入当前状态信息（缓存友好）"（可选）

### 3. 在 LLM 回复中使用状态更新命令

LLM 可以在回复中包含状态更新命令，格式为：

```
_.set("角色名.状态名",oldvalue,newvalue)
```

可以一次更新多个状态：

```
用户：你今天心情怎么样？

AI：我今天心情很好！_.set("角色名.好感度",50,55) _.set("角色名.信任度",60,65)
```

脚本会自动解析并更新所有状态。

### 4. 在世界书中使用状态变量

在世界书或其他地方，可以使用状态变量，格式为 `{{character_states.角色名.状态名}}`：

```
[当前状态]
好感度: {{character_states.角色名.好感度}}
信任度: {{character_states.角色名.信任度}}
```

这些变量会在 prompt 组合时被替换为实际的状态值。

## 状态定义格式

### JSON 格式

```json
{
  "name": "character_states",
  "characters": [
    {
      "name": "角色名",
      "states": [
        {
          "name": "状态名",
          "ranges": [
            {"min": 最小值, "max": 最大值, "content": "该区间对应的内容"},
            {"min": 最小值, "max": 最大值, "content": "该区间对应的内容"}
          ]
        }
      ]
    }
  ]
}
```

**字段说明**：
- `name`: 固定为 `"character_states"`
- `characters`: 角色数组
  - `name`: 角色名称（必须与当前角色名匹配）
  - `states`: 状态数组（每个角色可以有多个状态）
    - `name`: 状态名称
    - `ranges`: 数值区间数组
      - `min`: 区间最小值（包含）
      - `max`: 区间最大值（包含）
      - `content`: 该区间对应的内容文本

### 数值区间匹配规则

- 状态值会与所有区间进行匹配
- 找到第一个匹配的区间（`min <= 值 <= max`）后返回对应的content
- 如果状态值不在任何区间内，返回 `null`

## 配置选项

### 功能开关
- **启用角色状态管理**：总开关
- **在聊天末尾注入当前状态信息（缓存友好）**：是否在每次生成前，把当前状态值作为一条 system 消息追加到聊天末尾。末端追加不会污染 LLM API 的 prompt 缓存前缀，命中率不受影响。关闭后 LLM 仅看到状态定义（区间规则），看不到当前数值
- **开发模式（显示详细日志）**：是否在控制台显示详细的处理日志，用于调试

## 工作原理

1. **状态定义解析**：
   - 监听 prompt 组合完成事件
   - 从 prompt 中提取 `<character_states>` 标签内容
   - 解析 JSON 格式的状态定义
   - 根据当前角色名匹配对应的状态定义

2. **初始化**：
   - 新角色：如果状态变量不存在，初始化为0
   - 已有角色：如果状态变量已存在，跳过初始化
   - 支持初始化多个状态

3. **状态更新**：
   - 监听 `MESSAGE_RECEIVED` 和 `MESSAGE_UPDATED` 事件
   - 从消息文本中提取所有 `_.set()` 语句并计算哈希
   - **幂等性处理**：相同的状态更新语句（无论触发多少次）只处理一次
   - **智能去重**：只关注状态更新相关的部分，消息其他部分变化不影响处理
   - **回退机制**：当消息被编辑时，自动回退旧的状态更新，应用新的更新
   - 使用差值（delta）更新，避免 oldValue 不准确导致的问题

4. **静态定义替换（缓存友好）**：
   - 在 `CHAT_COMPLETION_PROMPT_READY` 事件中，把 prompt 中的 `<character_states>` 标签原地替换为**仅包含状态定义**的稳定文本（状态名 + 各区间 min/max/content），跨轮恒定
   - 不写入当前数值，避免每次状态变化时让前缀缓存失效

5. **当前状态末端注入**：
   - 在 `CHAT_COMPLETION_PROMPT_READY` 事件中，直接修改 `event_data.chat`，把"当前角色状态"作为一条 system 消息追加到**聊天末尾**
   - 由 `inject_current_state_at_end` 开关控制
   - 末端追加不会破坏 LLM API 的 prompt 缓存前缀
   - 状态信息格式：`[角色状态: 角色.状态名1 = 值1（min=… max=…）\ncontent1, …]`

5. **变量替换**：
   - 在世界书或其他地方使用 `{{character_states.角色名.状态名}}` 格式的变量
   - 酒馆会在 prompt 组合时自动替换这些变量为实际值

6. **历史消息初始化**：
   - 脚本加载时自动扫描所有历史消息
   - 记录每条消息的状态更新，但不重复应用（因为状态已经存在）
   - 确保编辑历史消息时能正确回退旧的状态更新

## 核心机制

### 1. 基于状态更新语句的幂等性处理

脚本使用**状态更新语句的哈希**来判断是否需要处理，而不是整个消息的哈希。这意味着：

- ✅ **精确去重**：相同的 `_.set()` 语句，无论触发多少次，都只处理一次
- ✅ **智能过滤**：消息其他部分的变化（如用户添加文本）不会触发重复处理
- ✅ **多脚本兼容**：即使多个脚本同时修改消息导致 `MESSAGE_UPDATED` 多次触发，也不会重复处理

### 2. 回退机制

当消息被编辑时，脚本会自动：

1. 检测状态更新语句是否变化（通过哈希比较）
2. 如果变化，先回退旧消息中的状态更新（使用差值的相反数）
3. 然后应用新消息中的状态更新

这确保了编辑消息时状态更新的正确性。

### 3. 差值更新

脚本使用 `delta = newValue - oldValue` 来计算更新量，然后应用 `currentValue + delta`。这样即使 LLM 提供的 `oldValue` 不准确，只要差值正确，更新就是正确的。

### 4. 并发处理保护

使用 `processing` 标志防止同一条消息被并发处理，确保状态更新的原子性。

## 注意事项

1. **状态定义位置**：`<character_states>` 标签可以放在角色描述、世界书或任何 prompt 位置
2. **状态更新命令**：LLM 需要在回复中包含 `_.set("角色名.状态名",oldvalue,newvalue)` 格式的命令才能触发更新
3. **变量格式**：在世界书中使用状态变量时，使用 `{{character_states.角色名.状态名}}` 格式
4. **自动解析**：状态定义会在每次 prompt 组合时自动解析和更新
5. **角色匹配**：状态定义中的角色名必须与当前角色名完全匹配
6. **区间重叠**：如果状态值同时匹配多个区间，会返回第一个匹配的区间content
7. **区间覆盖**：建议确保所有可能的数值都有对应的区间，避免返回null
8. **未闭合表达式**：如果消息被截断导致 `_.set()` 表达式未闭合，脚本会记录警告并忽略该表达式
9. **历史消息编辑**：编辑历史消息时，脚本会自动回退旧的状态更新并应用新的更新

## 优势

- ✅ **灵活的状态定义**：每个角色可以有多个状态，每个状态可以有多个数值区间
- ✅ **精细的状态控制**：不同数值区间对应不同的content，实现精细的状态控制
- ✅ **无需手动配置**：状态定义直接从 prompt 中解析，无需在设置界面中手动添加
- ✅ **自动初始化**：新角色自动初始化为0，已有角色保持原值
- ✅ **标准格式**：使用 `_.set()` 格式，与 lodash 语法一致
- ✅ **简洁的设置界面**：使用 SillyTavern 原生控件，界面简洁易用
- ✅ **智能去重**：基于状态更新语句的幂等性处理，避免重复更新
- ✅ **自动回退**：编辑消息时自动回退旧的状态更新，确保状态一致性
- ✅ **差值更新**：使用差值更新机制，即使 oldValue 不准确也能正确更新
- ✅ **多脚本兼容**：能够正确处理多个脚本同时修改消息的情况
- ✅ **缓存友好**：前缀区只放静态状态定义，当前数值仅追加在聊天末端，避免破坏 LLM API 的 prompt 缓存命中率
- ✅ **开发模式**：提供开发模式开关，方便调试和排查问题
